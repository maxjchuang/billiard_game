# 台球游戏（Demo）端到端功能测试用例（QA 版）

## 1. 文档目的

本文档面向 QA 端到端（E2E）功能测试，依据：

- `spec/billiard_game_technical_design.md`
- `spec/billiard_game_architecture_design.md`
- `spec/browser_debug_runtime_solution.md`
- `README.md`

目标是为 Demo 阶段建立“可执行、可复现、可留痕”的测试用例集合，用于：

- 验证 Demo 的核心闭环（启动 → 对局 → 结算 → 重开/返回）
- 回归关键逻辑：物理、规则、状态机、渲染、输入
- 用日志与截图定位问题，降低调试成本

## 2. 测试范围与优先级

### 2.1 P0（必须通过）

- 启动与基本页面流程：标题页 → 开始对局 → 结算页 → 重开/返回
- 一次完整出杆：瞄准、蓄力、出杆、球运动、球停止、结算
- 物理基础正确性：反弹、球-球碰撞、落袋、停球
- 规则基础闭环：首碰记录、落袋记录、犯规（至少覆盖母球落袋）、换手/续杆
- 关键链路日志完整且无异常（error 级别日志为 0）

### 2.2 P1（应通过）

- 多回合连续出杆（至少 5 回合）无状态异常
- 黑八胜负判定（合法/非法）
- 横屏适配与 UI/HUD 不遮挡球桌
- 切后台/恢复后对局状态不崩溃（微信开发者工具与浏览器）

### 2.3 P2（可选）

- 性能与稳定性：10 分钟压力跑动无明显卡顿/内存暴涨
- 异常网络/代理环境下浏览器调试入口可用性
- 低性能设备降级（后续阶段）

## 3. 测试环境

### 3.1 Web 调试环境（推荐日常回归）

- 启动：`npm run dev:web`
- 访问：`http://127.0.0.1:5173/`
- 目的：快速验证渲染、状态机、物理与规则闭环；可配合 `browser-use` 做半自动化验证

### 3.2 微信小游戏环境（上线前必须回归）

- 微信开发者工具导入项目
- 运行小游戏入口（`game.js` → `dist/game.js`）
- 目的：验证微信环境 Canvas、生命周期、输入事件差异与兼容性

## 4. 测试数据与通用前置条件

### 4.1 通用前置条件

- 已执行：`npm install`
- 已通过：`npm run lint`、`npm test`、`npm run build`
- 启用链路日志（默认 `ConsoleLogger`）

### 4.2 通用验收标准

- 不出现崩溃（页面白屏/卡死/脚本异常）
- 不出现未捕获异常（浏览器控制台 / 微信控制台）
- 对应链路日志必须可追踪（详见第 6 章）

### 4.3 留痕要求（执行测试必须遵守）

- 每次关键改动后的 E2E 回归，需要在多维表格记录：
  - `https://bytedance.larkoffice.com/base/FbKAbZJHFa60ZcsLj1vcIpIqn6g`
- 每条记录至少包含：
  - 测试范围（跑了哪些用例/回归点）
  - 结果（通过/失败、失败原因）
  - 截图（启动页/对局页/HUD/结算页）
  - 关键日志片段（如发生错误）

每条用例建议使用如下模板：

- 用例编号：E2E-XXX
- 优先级：P0/P1/P2
- 平台：Web/微信/双端
- 前置条件：...
- 步骤：1..N
- 预期结果：...
- 关键日志点：需要出现的 scope/message
- 截图点：需要截图的时刻

## 6. 链路日志检查点

本项目要求新增模块具备完整链路日志。E2E 执行时，需要确认下列核心链路日志出现且语义正确（以 scope/message 为主）：

- 启动链路
  - `GameSession/init`
  - `GameApp/boot-complete`
  - `SceneManager/set-scene`
  - `StateMachine/transition`
- 出杆链路
  - `ShotResolver/shoot`
  - `PhysicsWorld/step`
  - `PhysicsWorld/collision-ball`（出现球-球碰撞时）
  - `PhysicsWorld/ball-pocketed`（发生落袋时）
  - `RuleEngine/resolve`
  - `RoundResolver/round-resolved`
- 渲染链路
  - `Renderer/render-frame`
  - `TableRenderer/render`
  - `BallRenderer/render`

判定规则：

- `ERROR` 级别日志为 0
- `WARN` 允许出现，但必须可解释（例如某些平台能力缺失）

## 7. 端到端测试用例

> 说明：以下用例以“Demo 目标闭环”为主（见 `spec/billiard_game_architecture_design.md` 第 2 章）。部分交互（如触控拖拽出杆）若尚未实现，可先用开发调试入口/脚本方式触发，待 UI 完整后再替换为真实操作步骤。

### 7.1 启动与页面流程

#### E2E-001 启动进入标题页

- 优先级：P0
- 平台：Web/微信
- 前置条件：项目构建完成
- 步骤：
  1. Web：启动 `npm run dev:web`，打开 `http://127.0.0.1:5173/`
  2. 微信：开发者工具运行项目
- 预期结果：
  - 页面成功渲染，画面非白屏
  - HUD 顶栏显示标题与状态（例如 `state=menu`）
  - HUD 不遮挡球桌区域
- 关键日志点：`GameApp/boot-complete`、`SceneManager/set-scene`、`StateMachine/transition`
- 截图点：标题页整体画面（含 HUD 与球桌）

#### E2E-002 从标题页开始对局

- 优先级：P0
- 平台：Web/微信
- 前置条件：处于标题页
- 步骤：
  1. 点击“开始对局”按钮（若暂未实现按钮，则通过调试入口触发 start-match 意图）
- 预期结果：
  - 场景切换到对局
  - 球桌上出现球组（至少母球、目标球、黑八）
  - 状态机进入 `aiming`
- 关键日志点：`GameApp/start-match`、`SceneManager/set-scene`、`StateMachine/transition`
- 截图点：对局初始摆球

#### E2E-003 对局结束后进入结算页

- 优先级：P1
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 通过连续出杆使黑八落袋（合法或非法都可）
- 预期结果：
  - 触发 game over
  - 切换到结算场景
  - 显示胜负结果与重开/返回入口
- 关键日志点：`RuleEngine/resolve`（`gameOver=true`）、`RoundResolver/round-resolved`、`SceneManager/set-scene`
- 截图点：结算页

#### E2E-004 结算页重开与返回

- 优先级：P0
- 平台：Web/微信
- 前置条件：处于结算页
- 步骤：
  1. 点击“重开”
  2. 点击“返回标题页”
- 预期结果：
  - 重开后回到对局初始摆球
  - 返回后回到标题页
  - 状态机与场景切换无异常
- 关键日志点：`SceneManager/set-scene`、`StateMachine/transition`

### 7.2 出杆闭环

#### E2E-101 瞄准 → 蓄力 → 出杆 → 球运动 → 停球 → 结算

- 优先级：P0
- 平台：Web/微信
- 前置条件：对局中且允许出杆
- 步骤：
  1. 调整瞄准方向
  2. 调整力度
  3. 确认出杆
  4. 等待球全部停止
- 预期结果：
  - 母球获得初速度并运动
  - 若发生碰撞，目标球产生运动
  - 最终所有球速度归零
  - 触发回合结算并进入下一回合或结束
- 关键日志点：`ShotResolver/shoot`、`PhysicsWorld/step`、`RuleEngine/resolve`、`RoundResolver/round-resolved`
- 截图点：
  - 出杆瞬间
  - 球运动中
  - 停球后（结算完成）

### 7.3 物理正确性（基础）

#### E2E-201 球与库边反弹

- 优先级：P0
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 对准库边方向出杆，确保母球与库边发生碰撞
- 预期结果：
  - 发生反弹
  - 反弹后速度方向符合反射规律（允许能量损耗）
- 关键日志点：`PhysicsWorld/collision-rail`

#### E2E-202 球与球碰撞

- 优先级：P0
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 母球直线撞击目标球
- 预期结果：
  - 目标球获得速度并运动
  - 母球速度发生变化
  - 首碰记录为目标球
- 关键日志点：`PhysicsWorld/collision-ball`，并在 `PhysicsWorld/step` 结果中出现 `firstHitBallId`

#### E2E-203 落袋判定

- 优先级：P0
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 将任意目标球打入袋口
- 预期结果：
  - 球被标记为落袋并从桌面消失
  - 结算中包含该球的落袋记录
- 关键日志点：`PhysicsWorld/ball-pocketed`、`GameSession/mark-pocketed`

#### E2E-204 停球阈值与停球时机

- 优先级：P1
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 轻力度出杆
  2. 观察球速度衰减直至停止
- 预期结果：
  - 速度逐渐衰减
  - 低于阈值后速度被置零
  - 停球后触发结算
- 关键日志点：`PhysicsWorld/step` 中 `allStopped=true`

### 7.4 规则正确性（简化 8 球）

#### E2E-301 首碰为空/非法处理

- 优先级：P1
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 出杆后未碰到任何球（例如直接停下或进入袋口等）
- 预期结果：
  - 规则判定中出现 `no-first-hit`
  - 判定犯规并换手
- 关键日志点：`RuleEngine/resolve` 中 `foul=true` 且包含原因

#### E2E-302 母球落袋犯规

- 优先级：P0
- 平台：Web/微信
- 前置条件：对局中
- 步骤：
  1. 将母球打入袋口
- 预期结果：
  - 本杆犯规
  - 下一回合换手
  - 进入自由摆球（Demo 可先表现为 `cueBallInHand=true` 状态）
- 关键日志点：`RuleEngine/resolve` 包含 `cue-ball-pocketed`
- 截图点：母球落袋后结算状态

#### E2E-303 未分组前进球触发分组

- 优先级：P1
- 平台：Web/微信
- 前置条件：双方未分组
- 步骤：
  1. 打进一颗全色或花色
- 预期结果：
  - 当前玩家获得对应分组
  - 对手获得相反分组
- 关键日志点：`GameSession/assign-group`

#### E2E-304 黑八非法落袋判负

- 优先级：P1
- 平台：Web/微信
- 前置条件：当前玩家尚未清空自己分组目标球
- 步骤：
  1. 直接打进黑八
- 预期结果：
  - 判定非法黑八
  - 当前玩家直接判负
  - 进入结算页
- 关键日志点：`RuleEngine/resolve` 中 `illegal-black-pocket`、`gameOver=true`

#### E2E-305 黑八合法落袋判胜

- 优先级：P1
- 平台：Web/微信
- 前置条件：当前玩家已清空自己分组目标球
- 步骤：
  1. 合法击打黑八并落袋
- 预期结果：
  - 判定合法结束
  - 当前玩家获胜
  - 进入结算页
- 关键日志点：`RuleEngine/resolve` 中 `gameOver=true` 且 `winner=currentPlayer`

### 7.5 UI/渲染/适配

#### E2E-401 HUD 不遮挡球桌

- 优先级：P0
- 平台：Web/微信
- 前置条件：启动成功
- 步骤：
  1. 观察 HUD 与球桌区域
- 预期结果：
  - HUD 在顶栏
  - 球桌与球体绘制区域在 HUD 下方
  - 文本不覆盖球桌
- 截图点：启动页与对局页各一张

#### E2E-402 浏览器画布尺寸贴合球桌

- 优先级：P0
- 平台：Web
- 前置条件：启动成功
- 步骤：
  1. 观察页面画布尺寸
- 预期结果：
  - 画布尺寸与逻辑球桌 + HUD 一致（例如 640×(360+HUD)）
  - 页面不出现过大空白

### 7.6 生命周期与稳定性

#### E2E-501 切后台/恢复

- 优先级：P1
- 平台：Web/微信
- 前置条件：对局中球体运动中
- 步骤：
  1. Web：切换浏览器标签页 5 秒后切回
  2. 微信：模拟 onHide/onShow（或切换到其他页面再回来）
- 预期结果：
  - 不崩溃
  - 不出现状态机卡死
  - 恢复后可继续对局或重新开始

#### E2E-502 长时间运行稳定性（10 分钟）

- 优先级：P2
- 平台：Web/微信
- 前置条件：启动成功
- 步骤：
  1. 连续出杆并观察 10 分钟
- 预期结果：
  - 无明显卡顿
  - 无异常日志（error 为 0）
