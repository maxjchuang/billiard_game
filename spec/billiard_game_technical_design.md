# 微信小游戏台球游戏技术实现设计（Demo 方案）

## 1. 文档目标

本文档用于定义微信小游戏平台台球游戏的第一阶段 Demo 技术实现方案。当前阶段目标不是直接上线，而是在尽量可控的开发复杂度下，基于微信小游戏测试号先完成“可运行、可验证、可迭代”的单机台球原型，为后续正式版的 AI、联网、商业化与发布流程预留扩展空间。

## 2. 产品目标与版本范围

### 2.1 产品目标

- 打造一款运行于微信小游戏平台的 2D 台球游戏。
- 第一阶段聚焦原型 Demo，优先验证物理手感、规则闭环、横屏交互和运行稳定性。
- Demo 阶段以“能打通一局简化 8 球流程”为验收目标，不以正式上线为目标。

### 2.2 已确认产品决策

- 玩法：简化 8 球单机对局
- AI：首阶段不实现 AI 对手
- 屏幕方向：横屏
- 规则风格：偏真实台球规则
- 瞄准辅助：弱辅助
- 阶段目标：先做可玩 Demo
- 美术风格：极简干净风格
- 资源策略：先使用占位资源开发，后续替换正式素材
- 商业化：首阶段只做接口预留，不正式接入
- 技术路线：`TypeScript + 微信小游戏原生 Canvas 2D`

### 2.3 Demo 范围

Demo 阶段包含以下内容：

- 标题页
- 单机开始入口（可不做复杂模式选择）
- 对局页
- 结算页
- 基础重开与返回流程
- 占位资源下的球桌、球体、球杆与 UI
- 简化 8 球的基础规则闭环

Demo 阶段暂不包含以下内容：

- AI 对手
- 联网对战
- 复杂赛事系统
- 完整商业化系统
- 排行榜
- 云端存档
- 正式支付、广告、分享接入
- 正式上线所需认证、备案与提审流程
- 高级特效与复杂球体旋转表现
- 正式美术资源替换

## 3. 推荐玩法定义

### 3.1 首版玩法建议

建议 Demo 阶段采用“简化 8 球规则”，原因如下：

- 用户认知成本较低
- 规则相对成熟，便于形成完整对局体验
- 适合作为物理系统、裁判逻辑和回合系统的验证载体

### 3.2 核心体验流程

玩家在对局中完成以下操作：

1. 调整击球方向
2. 控制击球力度
3. 观察母球碰撞、目标球运动、反弹与进袋
4. 根据规则进行续杆、换手、犯规处理和结算

### 3.3 规则闭环

Demo 规则建议覆盖：

- 开球
- 分组（全色 / 花色）
- 合法击球判断
- 进球续杆
- 犯规判定
- 自由摆球
- 黑八结算

### 3.4 Demo 对局形态

当前阶段以“单机原型验证”为主，不实现 AI，也不接入联网。

- 默认由单设备完成整局流程验证
- 可以先按单人操作 + 裁判结算方式实现
- 若后续需要增强可玩性，再补本地双人轮流或 AI 对手

## 4. 技术选型

### 4.1 运行平台

- 平台：微信小游戏
- 语言：TypeScript
- 渲染：小游戏原生 Canvas 2D
- 平台能力：微信小游戏原生 API

### 4.2 选型原因

首版优先使用 `TypeScript + 原生 Canvas 2D`，原因如下：

- 包体更轻
- 运行链路更简单
- 更方便精细控制物理模拟细节
- 避免过早引入重型引擎增加适配和调优成本

若后续内容规模扩大，再评估引擎化方案。

## 5. 总体架构设计

建议采用分层架构，兼顾 Demo 交付速度与后续演进空间。

### 5.1 模块划分

- `core`：主循环、场景管理、调度器、事件系统
- `physics`：向量运算、碰撞检测、碰撞响应、运动积分
- `gameplay`：规则系统、回合控制、裁判判定、结算逻辑
- `render`：球桌渲染、球体渲染、UI 渲染、弱辅助线绘制
- `input`：触控输入、瞄准、力度控制、拖拽交互
- `platform`：微信 API 封装、存档、音频、震动、分享、广告预留
- `config`：规则参数、物理参数、资源表、常量
- `assets`：图片、音频、字体、预渲染贴图等资源

### 5.2 流程关系

主要链路如下：

1. `input` 收集触控操作
2. `gameplay` 根据当前状态解释输入
3. `physics` 执行球体运动与碰撞更新
4. `gameplay` 在所有球停止后执行规则结算
5. `render` 将当前游戏状态绘制到画面上
6. `platform` 负责与微信小游戏能力交互

## 6. 主循环与状态机设计

### 6.1 主循环

推荐使用固定时间步长的物理循环：

- 物理更新步长：`1 / 120s`
- 渲染频率：跟随屏幕刷新
- 物理更新与渲染解耦

这样做的好处：

- 降低不同机型帧率波动对物理稳定性的影响
- 让击球与碰撞结果更可预测
- 更容易做手感调优

### 6.2 状态机

推荐核心状态如下：

- `Boot`
- `Menu`
- `Aiming`
- `Powering`
- `BallsMoving`
- `RoundSettlement`
- `GameOver`

状态切换原则：

- 所有输入必须由当前状态决定是否可响应
- 所有回合结算必须在 `BallsMoving` 结束后触发
- 所有 UI 操作不能直接绕开状态机修改核心对局状态

## 7. 物理系统设计

### 7.1 球体数据模型

每个球建议维护以下数据：

- 唯一 ID
- 球类型（母球、黑八、全色、花色）
- 半径
- 质量
- 位置
- 速度
- 是否已进袋
- 可选：简化角速度或旋转表现参数

### 7.2 碰撞类型

首版需实现三类碰撞：

- 球与球
- 球与库边
- 球与袋口

### 7.3 碰撞策略

#### 球与球

- 使用二维圆形碰撞检测
- 碰撞响应采用近似弹性碰撞
- 使用法线方向速度分解计算碰后速度

#### 球与库边

- 使用边界反射模型
- 加入反弹能量损耗系数

#### 球与袋口

- 使用“落袋判定区 + 吸附区”方案
- 球进入袋口阈值区域后标记为进袋并触发下沉动画或直接隐藏

### 7.4 摩擦与停球

- 使用线性阻尼模拟台呢摩擦
- 当速度低于阈值时直接置零
- 所有球静止后才允许进入回合结算

### 7.5 首版物理取舍

Demo 阶段虽然规则风格偏真实，但物理实现仍采用“先闭环、后精修”的策略，不强求完全真实的旋转、塞球、杆法系统，先保证以下三点：

- 球路稳定
- 碰撞可信
- 手感连续

## 8. 瞄准与输入交互设计

### 8.1 输入方式

推荐采用触屏拖拽式交互：

- 拖拽调整瞄准方向
- 拉杆或力度条控制击球力度
- 抬手确认击球

### 8.2 瞄准辅助

Demo 阶段采用弱辅助，建议只保留最基础的瞄准参考：

- 母球出杆方向辅助线

不在 Demo 阶段实现：

- 首个目标球碰撞点预测
- 目标球碰后方向参考线
- 多次反弹预测
- 精细杆法预判
- 高级轨迹教学系统

### 8.3 操作状态约束

- 只有在 `Aiming` 或 `Powering` 状态允许控制球杆
- `BallsMoving` 状态必须禁用输入
- 确保连续点击、误触或切后台不会破坏物理状态

## 9. 规则与回合系统设计

### 9.1 核心对象

建议实现以下核心对象：

- `RuleEngine`：规则定义与合法性判断
- `TurnManager`：回合控制与换手逻辑
- `Referee`：犯规、进球、胜负判定
- `GameSession`：整局状态管理

### 9.2 回合结算流程

每轮出杆结束后：

1. 等待所有球停止
2. 统计进袋球
3. 判定首碰是否有效
4. 判定是否犯规
5. 判定是否续杆或换手
6. 判定是否胜负结束

### 9.3 配置化原则

规则中的以下内容建议配置化：

- 开球判定条件
- 犯规类型
- 落袋规则
- 黑八胜负条件
- 自由摆球范围

## 10. 渲染系统设计

### 10.1 渲染分层

推荐渲染层级：

1. 背景
2. 球桌
3. 球袋与装饰
4. 球影
5. 球体
6. 球杆与辅助线
7. UI
8. 弹窗与结算层

### 10.2 资源策略

- 球桌和 UI 使用图片资源
- 球体优先使用预渲染贴图
- 音效使用短音频资源，按需加载

### 10.3 屏幕适配

台球更适合横屏，Demo 阶段采用横屏方案：

- 以横屏为主
- 做安全区域适配
- 兼容不同分辨率和 DPR

## 11. 平台能力接入设计

### 11.1 微信小游戏能力封装

建议统一封装以下能力：

- 生命周期管理
- 本地存储
- 音频播放
- 震动反馈
- 系统信息获取
- 分享入口预留
- 激励视频广告预留

### 11.2 Demo 阶段开发与账号策略

当前项目按微信小游戏原型开发路径推进，先使用最基础的测试号完成开发验证。

- 开发阶段使用微信小游戏测试号与开发者工具进行本地调试
- 当前阶段不依赖正式发布能力，不接入正式审核、上线与商业化流程
- 当前阶段不在前端保存任何服务端密钥，仅在项目配置中使用测试号或后续正式 `AppID`
- 当 Demo 达到预期后，再补齐正式小游戏主体、认证、备案、提审与发布流程

### 11.3 存档内容

Demo 本地存档建议保存：

- 是否完成新手引导
- 基础设置（音量、震动开关）
- 历史最好成绩或胜场
- 已解锁模式

### 11.4 埋点建议

建议预埋以下关键行为：

- 进入游戏
- 开始对局
- 完成对局
- 犯规次数
- 中途退出
- 首局完成率

## 12. 性能与兼容性要求

### 12.1 性能目标

- 中低端机保持稳定 60 FPS 渲染体验
- 物理运算稳定，不出现明显穿模和抖动

### 12.2 优化方向

- 控制单帧对象分配，减少 GC 抖动
- 尽量复用向量对象与临时数组
- 渲染层避免高成本实时渐变计算
- 对低性能设备降级阴影、拖尾、特效

## 13. 推荐目录结构

```text
src/
  main.ts
  core/
  physics/
  gameplay/
  render/
  input/
  platform/
  config/
  assets/
```

推荐入口文件职责如下：

- `src/main.ts`：小游戏初始化与启动
- `src/core/`：主循环、状态机、场景切换
- `src/physics/`：运动与碰撞
- `src/gameplay/`：规则与对局逻辑
- `src/render/`：桌面、球体、UI 绘制
- `src/input/`：触控与击球输入
- `src/platform/`：微信能力封装
- `src/config/`：参数与资源配置

## 14. 开发里程碑建议

### M1：最小可运行原型

- 完成小游戏基础启动
- 渲染球桌
- 实现单球移动与边界反弹
- 跑通基础输入

### M2：核心对局闭环

- 支持多球碰撞
- 实现落袋
- 实现瞄准和力度控制
- 所有球停止后可结算

### M3：Demo 规则闭环版本

- 完成简化 8 球规则
- 完成犯规逻辑
- 增加结算页

### M4：从 Demo 走向正式版（后续阶段）

- 补齐音效、引导与完整 UI 表现
- 替换正式美术资源
- 评估认证、备案、提审与上线要求
- 按需要接入分享、广告或后端能力

## 15. 风险与重点关注项

### 15.1 物理手感风险

台球游戏的成败高度依赖手感，必须尽早建立可调参数体系，重点调优：

- 摩擦系数
- 反弹损耗
- 碰撞恢复系数
- 袋口吸附范围
- 力度曲线

### 15.2 规则复杂度风险

由于已确认规则风格偏真实，Demo 阶段需要重点控制以下平衡：

- 真实规则不至于压垮新手理解成本
- 回合切换清晰
- 犯规反馈明确

### 15.3 机型差异风险

不同设备刷新率与性能波动会影响物理稳定性，因此固定步长与降级策略必须优先验证。

## 16. 后续扩展方向

在 Demo 稳定后，可逐步扩展：

- AI 对手
- 闯关模式
- 联网对战
- 排行榜
- 球杆和球桌皮肤
- 活动与商业化系统

## 17. 当前结论

本项目 Demo 阶段应坚持以下策略：

- 玩法上先做简化 8 球单机对局，不实现 AI
- 规则上采用偏真实规则，但瞄准辅助保持弱辅助
- 技术上采用 `TypeScript + 微信小游戏原生 Canvas 2D`
- 研发上先用微信小游戏测试号完成 Demo 开发与验证
- 美术上先使用极简占位资源，优先跑通物理、规则与交互闭环

该方案可作为项目 Demo 开发、模块拆分和后续正式版演进的基础版本。
