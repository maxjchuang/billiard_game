# Feature Specification: dev:web 鼠标操作支持

**Feature Branch**: `001-web-mouse-control`  
**Created**: 2026-04-18  
**Status**: Implemented  
**Input**: User description: "dev:web 模式下需要支持鼠标操作"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 鼠标完成一次出杆 (Priority: P1)

作为在浏览器里调试台球玩法的玩家，我希望通过鼠标完成瞄准、蓄力、出杆，这样我不需要控制台命令就能直接体验核心玩法。

**Why this priority**: 这是 `dev:web` 模式最核心的可用性能力，缺失后无法进行真实交互验证。

**Independent Test**: 仅实现本故事时，用户可进入对局并通过鼠标完成一次合法击球，看到球体移动与回合推进。

**Acceptance Scenarios**:

1. **Given** 用户已在对局界面且球桌可交互, **When** 用户按下并拖动鼠标进行瞄准与蓄力后释放, **Then** 系统触发一次击球并进入球体运动阶段。
2. **Given** 用户完成一次击球, **When** 所有球停止, **Then** 系统完成回合结算并允许下一次鼠标击球操作。

---

### User Story 2 - 鼠标操作可预期且可取消 (Priority: P2)

作为玩家，我希望鼠标交互过程有明确反馈并支持中断，这样可以减少误操作。

**Why this priority**: 减少误击球会直接影响调试效率与体验稳定性。

**Independent Test**: 仅实现本故事时，用户可识别当前是否处于瞄准/蓄力流程，并可在出杆前取消本次操作。

**Acceptance Scenarios**:

1. **Given** 用户开始鼠标拖拽但尚未释放, **When** 用户取消本次拖拽, **Then** 系统不应触发击球且保持可继续瞄准。
2. **Given** 用户正在进行鼠标瞄准/蓄力, **When** 操作状态变化, **Then** 界面能反映当前状态并保持与实际行为一致。

---

### User Story 3 - 菜单与重开流程支持鼠标 (Priority: P3)

作为调试用户，我希望通过鼠标即可开始对局、重开对局或返回菜单，从而完成一轮端到端交互验证。

**Why this priority**: 该能力提高完整流程验证效率，但优先级低于核心击球交互。

**Independent Test**: 仅实现本故事时，用户无需键盘或控制台即可完成开始、重开与返回菜单操作。

**Acceptance Scenarios**:

1. **Given** 用户位于标题页, **When** 用户点击开始对局入口, **Then** 系统进入对局状态。
2. **Given** 用户位于对局或结算状态, **When** 用户点击重开或返回菜单入口, **Then** 系统执行对应状态切换并显示正确界面。

### Edge Cases

- 当用户在球体仍在运动时执行鼠标拖拽，不应触发新的击球。
- 当用户在球桌外按下或释放鼠标时，不应产生异常击球事件。
- 当用户快速连续点击或拖拽时，系统应避免重复出杆。
- 当用户拖拽距离极小或极大时，系统应将力度限制在允许范围内并保持稳定反馈。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在 `dev:web` 模式下支持通过鼠标完成瞄准、蓄力、释放出杆的完整交互。
- **FR-002**: 系统 MUST 将一次鼠标交互映射为单次击球意图，避免一次操作触发多次出杆。
- **FR-003**: 系统 MUST 在不允许击球的状态下拒绝鼠标出杆请求，并保持当前状态一致。
- **FR-004**: 系统 MUST 提供用户可感知的鼠标交互反馈，使用户能判断是否处于瞄准、蓄力或可出杆状态。
- **FR-005**: 用户 MUST 能在释放前取消当前鼠标操作，且取消后不产生击球结果。
- **FR-006**: 系统 MUST 在 `dev:web` 模式下提供可点击的开始对局、重开对局、返回菜单入口。
- **FR-007**: 系统 MUST 在异常鼠标输入（球桌外、快速重复、边界力度）下保持稳定，不崩溃且不产生非法回合推进。

### Key Entities *(include if feature involves data)*

- **Mouse Shot Interaction**: 描述一次从按下到释放（或取消）的鼠标击球会话，包含方向、力度、是否生效等结果属性。
- **Match Interaction State**: 描述当前对局是否接受鼠标击球输入，以及菜单/对局/结算等可点击入口的可用状态。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在 `dev:web` 模式中，90% 以上首次使用者可在 30 秒内完成第一次鼠标出杆。
- **SC-002**: 在 50 次连续鼠标出杆操作中，误触发多次出杆的比例为 0%。
- **SC-003**: 在 30 次边界输入测试（球桌外点击、快速连点、极小/极大拖拽）中，功能异常或流程中断次数为 0。
- **SC-004**: 在不使用控制台命令的前提下，用户可完成“开始对局 → 出杆 → 重开/返回菜单”完整流程，任务完成率达到 100%。

## Assumptions

- `dev:web` 已经是主要的浏览器调试入口，新增能力仅针对该入口，不改变微信小游戏运行方式。
- 本特性聚焦鼠标输入，不要求本次同时实现触屏手势输入。
- 现有回合判定与物理行为正确可复用，本特性仅补齐用户交互入口与输入映射。
- 当前版本的核心目标是可调试与可验证，不要求在本次交付中达到正式商业化 UI 质量。
