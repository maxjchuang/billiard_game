# Feature Specification: 物理参数实时 HUD 配置

**Feature Branch**: `008-physics-hud-config`  
**Created**: 2026-04-21  
**Status**: Draft  
**Input**: User description: "与物理特性相关的配置，做成一个可由用户在 UI 上进行实时配置的前端 HUD 界面"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 实时调节物理参数 (Priority: P1)

作为调试玩家或策划，我希望在前端 HUD 上直接查看并修改物理参数，这样我可以在不离开游戏画面的情况下立即观察手感变化。

**Why this priority**: 这是该需求的核心价值；如果不能直接在 HUD 中实时调节，就无法形成高效的参数试错闭环。

**Independent Test**: 进入可操作界面后，打开 HUD，修改任一受支持的物理参数，并在同一局或下一次击球中直接观察到结果变化。

**Acceptance Scenarios**:

1. **Given** 用户处于可见球桌与 HUD 的前端界面，**When** 用户查看物理配置面板，**Then** 用户可以看到每个受支持参数的名称、当前值和可调入口。
2. **Given** 用户正在使用物理配置 HUD，**When** 用户调整某个受支持参数，**Then** 新值会立即成为当前生效值，无需刷新页面或重新进入界面。
3. **Given** 用户已经修改某个物理参数，**When** 用户继续击球或观察球桌行为，**Then** 玩家能够感知到更新后的参数影响当前表现。

---

### User Story 2 - 调参过程安全可控 (Priority: P2)

作为调参用户，我希望 HUD 只允许输入合法范围内的物理参数，并在异常输入时给出明确反馈，这样我不会因为误操作把当前局面调到不可用。

**Why this priority**: 物理参数直接影响游戏行为；如果没有边界和反馈，实时调参会快速导致不可控体验。

**Independent Test**: 在 HUD 中尝试输入超范围值、非法值和边界值，确认系统始终保持配置处于可用范围，并给出清晰反馈。

**Acceptance Scenarios**:

1. **Given** 某个物理参数存在允许范围，**When** 用户输入超出范围的值，**Then** 系统会阻止或修正该值，并保持当前配置处于合法状态。
2. **Given** 用户输入空值、非数字或无效格式，**When** 系统接收该修改，**Then** 系统不会应用无效值，并向用户表明该输入未生效。
3. **Given** 用户正在连续调整多个参数，**When** 某一次修改失败，**Then** 其他已成功应用的参数保持有效，不会被意外回滚。

---

### User Story 3 - 快速恢复与对比默认配置 (Priority: P3)

作为调试玩家或策划，我希望能够快速恢复默认物理参数，并清楚知道当前值是否已经偏离默认值，这样我可以高效地做对比实验和回到稳定状态。

**Why this priority**: 参数调优通常需要多轮试验；没有恢复与对比能力，会显著增加回归成本和误判概率。

**Independent Test**: 先修改多个参数，再执行恢复默认操作，确认所有受支持参数都回到默认值，且界面反映恢复结果。

**Acceptance Scenarios**:

1. **Given** 用户已修改一个或多个物理参数，**When** 用户执行恢复默认操作，**Then** 所有受支持参数都会恢复到默认值并立即生效。
2. **Given** 某个参数当前值与默认值不同，**When** 用户查看 HUD，**Then** 用户能够明显辨认该参数已被修改过。
3. **Given** 所有参数都已恢复默认，**When** 用户再次查看 HUD，**Then** HUD 会显示当前配置已回到默认状态。

---

### Edge Cases

- 用户在球仍在运动时调整参数，系统仍需保持界面可操作且结果可预期。
- 某个参数调整到合法边界值时，系统仍需正常应用，不出现卡死、闪退或不可恢复状态。
- 前端界面空间较小或 HUD 折叠时，用户仍应能够访问全部受支持参数，而不遮挡关键操作信息。
- 当当前局面已经使用了旧参数计算结果时，后续修改只影响新生效后的行为，不应造成“显示值已变但 HUD 无反馈”的困惑。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在前端提供一个可见的 HUD 配置界面，用于展示和调整受支持的物理参数。
- **FR-002**: 系统 MUST 在 HUD 中展示每个受支持物理参数的名称、当前值、默认值和可调入口。
- **FR-003**: 系统 MUST 允许用户在前端界面内直接修改受支持的物理参数，而无需离开当前游戏界面。
- **FR-004**: 系统 MUST 在用户修改参数后立即使新值生效，且无需刷新页面或重新进入界面。
- **FR-005**: 系统 MUST 对每个受支持参数定义合法范围或合法选项，并防止无效值进入当前生效配置。
- **FR-006**: 系统 MUST 在用户输入非法值、空值或格式错误时明确告知该修改未生效。
- **FR-007**: 系统 MUST 支持用户一键将所有受支持物理参数恢复为默认值。
- **FR-008**: 系统 MUST 让用户能够辨认哪些参数当前值已偏离默认值。
- **FR-009**: 系统 MUST 在一次参数修改失败时保持其他已成功应用的参数继续有效。
- **FR-010**: 系统 MUST 保证 HUD 在正常使用过程中不会遮挡到使玩家无法完成主要调试或击球判断的关键信息。
- **FR-011**: 系统 MUST 将“受支持的物理参数”范围限定为当前产品允许用户实时调节、且会直接影响运动、碰撞、反弹、停球或落袋表现的参数集合。
- **FR-012**: 系统 MUST 在 HUD 中反映当前配置是否处于“默认状态”还是“已修改状态”。

### Key Entities *(include if feature involves data)*

- **物理参数（Physics Parameter）**: 一个可由用户实时调整的配置项，关键属性包括显示名称、当前值、默认值、合法范围/合法选项、是否已被修改。
- **物理参数 HUD 状态（Physics HUD State）**: HUD 当前展示与交互状态，关键属性包括是否展开、当前可见参数列表、每个参数的校验结果、整体是否处于默认状态。
- **参数变更记录（Parameter Change Event）**: 一次由用户触发的配置修改动作，关键属性包括目标参数、输入值、是否应用成功、失败原因（若有）。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户进入可调试界面后，可在 10 秒内找到物理参数 HUD，并完成至少 1 个参数的有效修改。
- **SC-002**: 在受支持的物理参数集合中，100% 的参数都可通过 HUD 查看当前值、默认值并完成修改。
- **SC-003**: 在非法输入、超范围输入和边界值输入三类场景下，系统 100% 保持当前配置处于合法状态。
- **SC-004**: 用户在完成多轮调参后，可在 3 秒内通过恢复默认操作回到默认配置状态。

## Assumptions

- 本需求面向带有前端调试能力的运行界面，用户可以直接看到 HUD 并进行交互。
- 首版范围仅覆盖“实时查看、修改、校验、恢复默认”这条调参闭环，不要求提供长期保存、导入导出或多人共享配置能力。
- 默认配置已在产品中存在，且每个受支持参数都已有明确的默认值与可接受范围。
- 若参数修改发生在对局进行中，用户接受“新值对修改后产生的行为生效”的实时调参模式。
