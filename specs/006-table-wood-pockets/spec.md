# Feature Specification: 台球桌木边与六洞可视化

**Feature Branch**: `006-table-wood-pockets`  
**Created**: 2026-04-21  
**Status**: Draft  
**Input**: User description: "台球桌的边缘应该是木头做的，而且有六个球洞，当前 UI 上都看不出来，需要实现"

## Clarifications

### Session 2026-04-21

- Q: 坐标体系对齐（渲染 vs 物理）采用哪种方案？ → A: B（定义共享 `railThickness`，物理边界/球洞中心与渲染统一使用同一套 table layout）
- Q: 球洞的视觉层级如何处理？ → A: A（球洞在桌面层绘制，位于球的下方；不做“遮罩/吞球”效果）
- Q: 木边厚度（`railThickness`）默认取值？ → A: 20
- Q: 球洞中心点是否需要内缩？ → A: A（按 `railThickness` 内缩，保证洞口完整可见）
## User Scenarios & Testing *(mandatory)*

### User Story 1 - 桌面外观一眼可辨 (Priority: P1)

玩家打开游戏时，能够一眼看出球桌的木质边缘与 6 个球洞位置，从而理解击球目标与边界。

**Why this priority**: 当前 UI 无法感知桌面结构，会直接影响玩家对规则与操作的理解。

**Independent Test**: 打开游戏进入可打球的场景后，不进行任何操作即可目视确认木边与 6 个球洞。

**Acceptance Scenarios**:

1. **Given** 玩家进入可看到球桌的界面，**When** 玩家观察桌面，**Then** 桌面外围呈现“木质边缘”视觉效果，且与台呢区域区分明显。
2. **Given** 玩家进入可看到球桌的界面，**When** 玩家观察球桌四周，**Then** 能清晰辨认出 6 个球洞（四角 + 两侧中点）。

---

### User Story 2 - 球洞位置与落袋结果一致 (Priority: P2)

玩家瞄准球洞击球时，球洞的视觉位置与游戏的落袋判定一致，避免“看起来进了但没进/没进但算进”的违和感。

**Why this priority**: 视觉与结果不一致会破坏信任感与可玩性，属于高感知问题。

**Independent Test**: 让任意球沿直线进入角袋/中袋，观察球进入洞口的视觉表现，并核对落袋结果。

**Acceptance Scenarios**:

1. **Given** 玩家朝角袋击球，**When** 球进入角袋洞口区域，**Then** 球的视觉运动与落袋结果一致（进入洞口后产生明确的“入袋”反馈）。
2. **Given** 玩家朝中袋击球，**When** 球进入中袋洞口区域，**Then** 球的视觉运动与落袋结果一致。

---

### User Story 3 - 不同屏幕/窗口下都清晰可见 (Priority: P3)

玩家在不同分辨率或窗口尺寸下游玩时，木边与球洞仍保持清晰可见，不被 UI 元素遮挡。

**Why this priority**: 该项目存在不同运行环境与尺寸差异，桌面结构需要稳定可辨识。

**Independent Test**: 在不同窗口尺寸下进入游戏界面，确认木边与 6 洞都可见且不被遮挡。

**Acceptance Scenarios**:

1. **Given** 玩家在较窄窗口中进入游戏，**When** 观察桌面，**Then** 仍能辨认木边与 6 个球洞。
2. **Given** 玩家在较宽窗口中进入游戏，**When** 观察桌面，**Then** 木边与 6 个球洞布局不变形到难以辨认。

---

### Edge Cases

- 窗口尺寸极端（很宽/很窄）时，球洞仍应可见且位置正确。
- 画面上存在其他 UI 覆盖层时，不应遮挡球洞关键区域。
- 颜色对比不足导致球洞难以辨认时，需要仍能一眼分辨“洞口”与“台呢/木边”。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 在球桌台呢区域外围呈现可辨识的“木质边缘”视觉效果。
- **FR-002**: 系统 MUST 呈现 6 个球洞的清晰视觉元素（四角 + 两侧中点），且能与周围区域明显区分。
- **FR-003**: 系统 MUST 保证球洞在正常游玩视角下不会被常用 UI 元素遮挡。
- **FR-004**: 系统 MUST 保证球洞的视觉位置与游戏的落袋判定在玩家观感上保持一致。
- **FR-005**: 系统 MUST 在项目支持的运行环境与常见屏幕尺寸下保持上述效果可见且稳定。
- **FR-006**: 系统 MUST 使用一套共享的球桌布局定义（包含木边厚度、台呢区域、球洞几何），同时驱动物理边界/落袋判定与渲染展示，避免坐标体系不一致。
- **FR-007**: 系统 MUST 在首版以“桌面层绘制球洞、位于球的下方”为默认呈现方式；不要求实现球洞遮罩或落袋动画效果。
- **FR-008**: 系统 MUST 将默认木边厚度 `railThickness` 设为 20，并以此派生台呢区域与球洞位置（作为对齐渲染与物理的基准）。
- **FR-009**: 系统 MUST 将 6 个球洞的中心点按 `railThickness` 内缩到木边内侧，确保洞口视觉完整且与落袋判定一致。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 玩家进入游戏后的 5 秒内，可以正确指出“木边”与 6 个球洞位置（无需提示）。
- **SC-002**: 在常见窗口尺寸下，玩家均可清点出 6 个球洞且不被 UI 遮挡。
- **SC-003**: 进行落袋操作测试时，球进入洞口的视觉反馈与落袋结果一致（不出现明显违和的“进了不算/没进算进”情况）。

## Assumptions

- 默认采用标准 6 洞球桌布局：四角袋 + 两侧中袋。
- 本需求聚焦“可视化呈现与一致性”，不要求新增或改变规则体系。
- 桌面结构的可见性优先于装饰细节（先“看得出来”，再“更真实”）。
- 台呢区域为“木边内缩后的可玩区域”，并与物理世界的边界/球洞中心保持同源一致。
